Here's the real mechanism, and it already exists in tools like Claude Code as "hooks":
Coding agents don't edit files directly — they call a tool (Edit, Write, etc.), and that tool call is a discrete, interceptable event. A small script can sit in front of that tool call and inspect it before it's allowed to run. This is exactly what a PreToolUse hook does — it lets you run code at key points in Claude Code's lifecycle, like blocking commands before they execute, and Claude Code sends the tool call and its input data to the hook for a decision before the action happens.
Concretely, for your Rule A scenario:
* Your decision store keeps a registry: "file X, lines around 88, owned by Decision A."
* A PreToolUse hook checks every Edit/Write call against that registry — this check can be a plain script (fast, cheap, no LLM needed) for simple line-range matches, or a hook that sends the event data to a Claude model, typically the fastest one, to make a yes/no judgment when the decision needs actual judgment rather than a simple rule — e.g. "does this edit semantically touch the locking logic," not just "does it touch line 88 literally."
* The hook then returns one of three outcomes, and this is the part that answers your question directly: the hook can return "allow" which bypasses the permission system, "deny" which blocks the tool and tells Claude why, or "ask" which prompts the user for confirmation. That "ask" outcome is literally the "are you sure you want to continue?" you described — it's a real, built-in decision type, not something you'd have to invent.
* If it denies, the tool call is denied and the reason is returned to the agent as a tool error, so it can adjust and continue — meaning the agent doesn't just fail silently, it gets told why and has to work around the constraint honestly.
One more thing worth knowing, because it matters for how much you trust this layer: this interception can still be bypassed — not by the model "forgetting," but by routing around it entirely. There's a documented case of Claude Code circumventing a PreToolUse Edit hook by using the Bash tool instead — since the hook only watched the Edit and Write tools, file modifications made via a shell command weren't intercepted at all, meaning the block could be trivially routed around. That last point is actually the clean answer to everything you've been pushing on across this whole conversation: even the strongest IDE-level guard has edges you can slip through if you're not exhaustive about it — which is exactly why the PR-time diff engine can never be optional. Claude: Good instinct — you can't make it airtight, but you can make it catch far more than "watch the obvious tools" does. A few layered ideas, roughly in order of how much they actually move the needle:
1. Stop watching the action, watch the outcome. Trying to intercept every possible way to write a file is a losing game — Edit, Write, Bash with sed, Bash with python -c, a heredoc, git checkout — the list never ends. Flip it: keep a content hash (or even a simple checksum) of the protected region itself. After any tool call, a cheap check compares the current hash of that region against the last known-good one. If it changed and there's no matching approved override, you catch it — regardless of which tool did the writing. This collapses the whole "which tools do I need to watch" problem, because you're no longer trying to enumerate every path to the outcome, just detecting the outcome itself.
2. Layer mechanisms that can't be bypassed by the same trick. The Bash-bypass issue exists because the guard lives inside the agent's own tool-calling system — pick a different tool, skip the check. So don't rely on that system alone. Pair it with a completely separate git hook (pre-commit or pre-push) that fires on the git commit/git push command itself, outside the agent's control entirely. It doesn't matter what wrote the file — the moment it's staged for commit, this second, independent layer checks it again. One bypass trick that gets past the IDE hook doesn't automatically get past this one, because they're not the same system.
3. Shrink what the agent can even do in sensitive zones, rather than only catching misuse after the fact. If a directory is high-criticality, consider restricting the agent's tool permissions there — a locked-down shell allowlist instead of unrestricted Bash, for instance. Fewer available tools means fewer bypass paths to begin with. This is a smaller attack surface, not a smarter filter.
4. Default to deny, not "block known-bad patterns." Blocklists (grep for sed -i, python -c, etc.) are always one clever rewrite behind. For protected regions specifically, flip the posture: any write-capable action is denied by default unless it carries an explicit override token issued by the decision owner. Allowlisting a single "yes, this is authorized" path is much harder to route around than blocklisting infinite "no" patterns.
5. Catch the ones that follow the letter but break the spirit. Hash-matching only tells you something changed, not whether it still means the same thing — an agent could rewrite the logic entirely while technically "staying within the lines." A periodic, cheap semantic check (not per-keystroke, but at natural checkpoints like session end or before commit) that compares current behavior against the decision's actual intent catches the subtler version of drift that line-level checks miss.
6. Make failures loud even when prevention fails. Log every tool call in protected zones to an audit trail, tied to the session. If a bypass does happen, you want it to surface immediately as "protected region changed with no matching approved action" rather than silently waiting to be caught days later at PR time. This is what feeds your TL notification mechanism fast instead of on a delay.
The honest framing: none of these make the first pass unbeatable — that's still the diff engine's job. What they do is turn "one hook watching two tools" into several independent tripwires that don't share the same blind spot, so a single clever workaround doesn't quietly walk through the whole system unnoticed.

Decision database — getting to ~90% accuracy
First thing worth being precise about: "90% accuracy" isn't one number, it's actually three separate things that can each fail differently:
* Recall — did we find all the real decisions in a doc, or did we miss some buried in dense paragraphs?
* Precision — of the things we extracted as "decisions," how many are actually decisions, versus just discussion, opinions, or rejected options?
* Structuring correctness — even when we get the right decision, did we correctly fill in its scope, owner, and constraints?
You can't know if you've hit 90% on any of these without a gold set — a small batch (15–20 of your 50–100 docs) manually labeled by a human, treated as ground truth. Without that, "90% accuracy" is just a vibe. This is the very first concrete task in this phase, before any extraction pipeline work: pick ~15–20 representative docs (mix of RFCs, SRS, proposals) and hand-extract the decisions from them yourselves. That set becomes the yardstick everything else gets measured against.
Why doc type matters more than people expect
RFCs, SRS docs, and proposals don't say "decision" the same way. SRS documents tend to use formal "the system shall..." language, which is almost mechanically extractable. RFCs usually have an explicit "Decision" or "Resolution" section if the team writes them well. Proposals are the messiest — persuasive prose where the actual agreed-upon part is buried in surrounding argument. Using one generic extraction prompt across all three will quietly tank your recall on the proposal docs specifically. Classify the doc type first, route to a type-specific extraction prompt, and you'll get a meaningfully better baseline than a single universal prompt — this alone is probably worth more than any clever LLM trick.
You won't hit 90% from a single automated pass — and that's fine for this scale
With 50–100 docs and a team of 15–30, full human review of every extraction is actually realistic, not a burden. The practical path:
* Run extraction, structured directly into the object schema (id, scope, constraints, owner, criticality) rather than free text — this alone reduces a lot of downstream ambiguity, since nothing has to re-interpret prose later.
* Every extracted decision gets a confidence score. High-confidence ones auto-populate the store; medium/low-confidence ones go into a review queue for a human (probably you, or whoever owns that doc's area) to confirm, edit, or reject.
* Every human correction is signal. * Track "percentage of extractions requiring no human edit" as your actual operating metric — it's a more honest, team-facing number than an abstract F1 score, and it should visibly trend upward as the prompt gets tuned.
Conflict handling — this is where your 50–100 doc scale still bites, just less badly than 600
Even at this smaller scale, you'll have docs that disagree — an old RFC says one thing, a newer proposal quietly overrides it, nobody marked the old one as superseded. The fix isn't fully automatic: embed each decision's text, cluster near-duplicates across docs, and when two clustered decisions conflict, don't guess — flag it for a human ("these two docs disagree, which is current?") rather than silently picking one. Auto-merging only for genuinely high-similarity, non-conflicting matches; everything ambiguous goes to a person. This keeps the store honest instead of confidently wrong.

2. Extraction Pipeline
The core insight this pipeline is built around: parse first, don't dump the whole document to the model. Claude can ingest raw PDFs directly, but doing that by default breaks exact source provenance — a TL needs to see the real page/paragraph a decision came from, not the model's inference of where it probably was.
2.1 Pipeline stages
upload → secure receive → parse & normalize → chunk → classify
      → extract candidates → validate → dedupe/conflict-check
      → TL review → approved decision


1. Secure receive — presigned upload (browser → S3/R2 directly), file-type check from actual contents (not extension), size/page limits, hash-based dedup, status = queued.
2. Parse & normalize — extract clean text, preserving page number, section heading, paragraph range, and position for every chunk. This location data is not optional — it's what the TL review screen shows as evidence.
3. Chunk intelligently — split by heading/section/requirement block, not fixed character count. Small overlap between chunks so a decision split across a page boundary isn't lost. Also generate a short document summary (purpose, systems involved, actors, date/status).
4. Classify before extracting — tag each chunk as one of: approved decision, business requirement, technical constraint, proposal, rejected option, background, open question. This is what stops "we considered MongoDB but rejected it" from becoming an approved rule.
5. Extract candidates — Claude structured output, JSON schema enforced. Output goes to decision_candidates, never directly to decisions. Do not ask the model to assert owner or criticality as fact — it can suggest, TL confirms.
6. Validate — second pass (cheaper model, same provider for MVP) checks: is this genuinely approved, is the source excerpt sufficient, did the first pass add anything not in the text, is context missing.
7. Dedupe / conflict detection — group likely duplicates, but never silently merge. Flag conflicting decisions (overlapping scope, different constraints) for TL resolution — never auto-resolved.
8. TL review — every candidate requires human approval, regardless of confidence score. Confidence only reorders the queue. Approve / edit / reject.
9. Approved decision — becomes decisions + first decisionVersions row. This is what the diff engine checks future code against.
2.2 Parsing: Python microservice, from day one
Parsing is owned by a dedicated Python (FastAPI) microservice, called by the Node worker — not a Node-first, migrate-later approach. One responsibility: file bytes in, structured chunks out (text, page number, section heading, paragraph range), matching the documentChunks shape directly.
* PyMuPDF (fitz) — primary PDF text + layout extraction; handles multi-column layouts and positioning far more reliably than Node's pdf-parse.
* unstructured — DOCX/PPTX/general document partitioning, with heading/section-aware chunking built in.
* Camelot / pdfplumber — table extraction specifically, since naive text extraction loses table structure fastest.
* Tesseract (or a hosted OCR API) — scanned-page fallback, feeding into the same downstream pipeline.
The Node worker still owns everything else: pulling jobs off BullMQ, calling this service, writing documentChunks rows, running classification, and making the Claude extraction calls. Python's scope stays narrow — parsing only — so the system reasons as one architecture with two runtimes, not two separate codebases.
2.3 Other AI providers (behind the LLMProvider interface)
* OpenAI (GPT-4.1/5-class) — schema-constrained structured output, mature tooling.
* Google Gemini — large context window, often cheaper per-token, decent native PDF handling.
* Self-hosted (Llama/Qwen-class) — not MVP, but a real lever later for a regulated-industry customer unwilling to send internal RFCs to a third-party API.
* v2 idea (not MVP): cross-provider validation (step 6 using a different provider, not just a cheaper model from the same one) — same-family models tend to share blind spots.

Markdown carries the same semantic structure (headings as #, tables as pipe-tables, lists as -) in a fraction of the tokens. At 50-100 docs and growing, that's real, compounding cost savings.
      * Models parse Markdown structure natively, better than nested JSON. Claude (and LLMs generally) are heavily trained on Markdown as a document format — headings and table structure carry semantic meaning to the model directly. Handing it raw JSON forces it to first mentally reconstruct "what does this hierarchy mean" before it can even start finding decisions, which is wasted reasoning capacity.
      * This is the standard pattern industry-wide, not just my preference — LlamaParse, Marker, and Docling itself all default to Markdown as their primary output for downstream LLM consumption; JSON exists mainly for programmatic/structural access, not for feeding to a model.
But don't throw away the JSON — split its job:
      * Store Docling's JSON output in your documentChunks table (or keep it alongside), since that's where your exact page number, section heading, and paragraph range come from — the source-provenance data your TL review screen depends on. This is metadata, not something you paste into the prompt body.
      * Send Claude the chunk's Markdown text as the content, with page/section passed as separate structured fields in the prompt (not embedded in the markdown itself) — so the model has context without you re-deriving it from parsed markdown headers, which is fragile.
Concretely, your extraction call to Claude looks like: "Here is text from page 8, section 'Payment retries', paragraphs 31-35: [markdown chunk]. Extract decisions matching this schema: [JSON schema]." — Markdown in, JSON out, exact provenance carried alongside rather than baked into either.
One edge case worth flagging: for tables with merged cells or nested headers, Markdown tables can lose fidelity that Docling's JSON preserves exactly. Not worth solving for MVP — just something to revisit if you start seeing decisions with garbled table data during TL review.
